### 1-2. 仕組み（入退室あいさつのフロー）

1. メインループが `pollMembersAndDiff()` を毎周回呼ぶ（中で 12 秒スロットル）。
2. `yay_api.py members <call_id>` で現在の参加者名簿を取得。`callId` は `process.env.YAY_CALL_ID || creds.conference_id`。
3. 前回の roster と差分を取る：
   - 名簿に増えた人 = **join** → `jingleQueue` に積む
   - 名簿から消えた人 = **leave** → 同上
4. `drainJingle()` がキューを 1 周回 1 件・最小 8 秒間隔で消化。チャットに `🎉 〇〇 やっほー！いらっしゃい` を送信＋`sayJingle()` で TTS 読み上げ（音楽は止めず ttsGain に重ね、声の間だけ音楽をダッキング）。

**賢くしてある点（誤発火対策）:**
- `memberSeeded`: 起動直後の初回ポーリングは**無言でシード**（既に居る人に連打しない）。
- 5 秒以内の瞬断（回線フラップ）は無視（`gap < 5000`）。
- 90 秒以内の再入室は「おかえり」扱い（`rejoinGraceMs`）。
- 朝/昼/夜で挨拶語が変わる（`greetWord()`: おはよ／やっほー／こんばんは）。
- `quietHours`（例 `[23,7]`）を設定すれば深夜は声を出さず文字だけ（既定 null=常時声OK）。

### 1-3. 受け側（agora）は元から対応済みだった

`yay_music_bot/lib/agora.mjs` と `agora_client.html` には、分離時（2026-06-06）から **TTS 配信路がそのまま残っていた**（`playTTS` / `fileUrl` / `startFileServer` ＋ HTML 内の「音楽＋TTS ミックスバス・ダッキング」）。CONCEPT では「TTS やらない」と書いてあったが配管だけ生きていたので、それを再利用しただけ。受け側の追加実装は不要だった。

## 2. bot の全体像（現状できること）

### コマンド一覧（`/h` で表示される実物）
```
🎧 音楽BOT コマンド
/p <曲名/URL> = 再生（空きなら即/再生中はキューへ）
/q <曲> = キュー追加 / /ql = キュー一覧
/s = スキップ / /x = 停止（キュー消去）
/ps = 一時停止 / /r = 再開 / /l = 一曲ループ
/v 0-100 = 音量（既定15）/ /vu /vd = ±10
/np = 再生中 / /qd <N> 削除 /qu <N> 前へ /qj <N> 後ろへ /c 消去
/lv [入力] = システム音声配信 / /d = 入力一覧 / /bye = 退出
/greet [on/off] = 入退室の読み上げ（既定ON・名前で短く挨拶）
※「○○かけて」など自然言語でも再生できる
```
- 音源は **YouTube 一本**（`yt-dlp` で解決 → Agora RTC に publish）。
- キュー機能フル（追加・一覧・削除・前後並べ替え・自動送り）。
- 自然言語の音楽指示（「○○かけて」「止めて」「次」「音量上げて」等）を `nlToCommand()` で決定論変換。通話の全員が操作可。

### コンテキスト消費ゼロの理由（重要・壊すな）
- `claude -p` / Anthropic API を**一切呼ばない**。あいさつ文も「決め打ち候補からランダム選択」で AI 不使用。
- コンテキストを食うのは「会話返信・自発おしゃべり・曲をAIに選ばせる判断」の 3 つだけで、この bot はそれを**載せていない**。
- **新機能を足す時もこの原則を守ること。**「いい感じに喋らせる」等で `claude -p` を呼び込むと設計が崩れる。

## 4. 設定（`config.mjs` の `jingle`）

```js
jingle: {
  enabled: true,        // 既定ON（/greet off で停止）
  voice: true,          // 声でも読む（false=チャット文字だけ）
  voiceKey: null,       // TTSボイス（tts.mjs の VOICE_PACKS キー。null=エンジン既定。VV生きてれば speaker3=ずんだもん、無ければ say Kyoko）
  pollMs: 12000,        // 名簿ポーリング間隔（小さくすると反応速いが python spawn 頻度増）
  minGapMs: 8000,       // 連続あいさつの最小間隔
  maxQueue: 8,          // あいさつ待ち行列の上限
  rejoinGraceMs: 90000, // この時間内の再入室は「おかえり」
  quietHours: null,     // 例 [23,7] で深夜は無声（文字だけ）。null=常時OK
}
```
- 反応をもっと速くしたい（最大12秒遅延を縮める）→ `pollMs` を下げる。
- 声を中立にしたい（ずんだもん以外）→ `voiceKey` を `'sora'`/`'akari'` 等（`lib/tts.mjs` の `VOICE_PACKS` 参照）or `'say_default'`（macOS Kyoko 固定）。

## 5. 設計判断の記録（蒸し返さないこと）

### Spotify 連携 → 見送り確定（2026-06-10、究判断）
- **要望の流れ**: 音楽bot/フル機能bot どちらでも Spotify で流したい → でも見送りに。
- **技術的事実**: Spotify はフル音源を DRM で保護し、データ(ファイル)として渡さない。よって bot で流すには「ブラウザで実際に再生 → その音を BlackHole(仮想オーディオ)経由で拾って通話へ」という**音声配線が必須・回避不可**。（YouTube/yt-dlp はデータ直取りなので BlackHole 不要 → だから今の bot は BlackHole 無しで動く。）
- **究の判断**: 作業環境に仮想オーディオを常駐させたくない／「BlackHole 無しで鳴らせないなら要らない」→ 見送り。
- **物理パージも見送り**: BlackHole 2ch のアンインストール＋「Yay出力」集約デバイス削除も検討したが、再起動・sudo・Audio MIDI Setup 手動が絡む割に益が薄く、**何も消さず放置**を選択。
  - 既存のまま放置するもの: `BlackHole 2ch`(brew cask, 0.6.1)、`Yay出力`(Audio MIDI Setup の集約デバイス)、`yay_bot` の眠っている Spotify コード（`lib/music.mjs`, `scripts/launch_music.sh`, `bot_agora.mjs` の startSpotify/startAny 等、`config.spotifyEnabled:false` で無効化済み）。使わなければ無害。
- **結論**: この bot に Spotify は**入れない**。次セッションで再提案しないこと。参照 memory: `project_yay_music_bot_2026-06-06.md`。

## 6. 残タスク

### 音楽機能の差は解消済み（2026-06-10、追加コミットで実装）
フル機能 `yay_bot` にあって music_bot に無かった音楽機能を全部移植した（Spotify を除く）：
- **`/qr`（qrepeat）キュー全体リピート** … `queueRepeat` 状態 + 自動送りで流した曲を末尾へ戻して循環。
- **`/qsh`（qshuffle）シャッフル** … Fisher-Yates でキューを並べ替え。
- **複数曲一括追加** … `/q 曲A, 曲B`（カンマ/読点/改行区切り、`splitSongs`）。
- **YouTubeプレイリストURL展開** … `list=` 付きURLを `music.expandPlaylist`（`lib/music_agora.mjs` に移植）で全曲キューへ（上限 `PLAYLIST_LIMIT=100`）。
- 付随: `renderQueue` を `qLabel`（URL短縮）＋先頭15件表示＋リピート状態🔁表示に更新。`statusLine` にも🔁。

**これで Spotify 以外の音楽機能は yay_bot と同等。** 残タスクは無し（Spotify は §5 のとおり見送り確定）。
